# /a11y

Audit a component or page for accessibility. Run this before any component is considered complete.

Accessibility support is split into two tiers:

**Tier 1 — Required now.** Costs almost nothing, benefits everyone including sighted users on keyboards, outdoor mobile users, and people on slow devices. Blocking — a component does not pass `/a11y` without these.

**Tier 2 — ARIA / screen reader support.** Deferred to V4/V5, prioritised when user feedback requests it. Web Component shadow DOM makes ARIA non-trivial (cross-boundary `aria-labelledby` breaks, and not all AT tools handle intra-shadow live regions consistently). For the current use cases (sports scoring, event capture, field apps) the overlap with screen reader users is low. The foundation below keeps ARIA retrofittable without restructuring. Mark findings as advisory — do not block completion.

## Usage
/a11y <filepath>

Example: `/a11y core/components/update-banner/update-banner.js`

## Tier 1 — Required (blocking)

### Semantic structure

- Does the host element use a native HTML element where possible? (`<button>` not `<div onclick>`, `<nav>` not `<div class="nav">`)
- Are landmark regions present on page components? (`<nav>`, `<main>`, `<header>`)
- Are headings in logical order with no skipped levels?
- Is reading order in the DOM logical without relying on visual position?

### Interactive elements

- Does every button, link, and control have a visible label? Icon-only buttons must have a visible tooltip or adjacent text, even if small.
- Are all interactive elements reachable by keyboard (`Tab`)? Custom elements must have `tabindex="0"` on host or `delegatesFocus: true` on shadow root.
- Do all interactive elements respond to `Enter` and `Space` as well as pointer events?
- Does focus move logically through the page? No accidental focus traps.

### Touch targets

- All interactive elements meet `--touch-target` (40px) minimum. Primary actions meet `--touch-target-lg` (52px). Verify both dimensions.
- Swipe gestures and drag interactions have a single-tap alternative. No gesture can be the only path to an action — this affects sighted users with motor difficulties too, not just screen reader users.
- The drag-to-complete button must trigger on a single tap as a fallback.

### Colour and contrast

Check against WCAG AA minimums:
- Normal text (below 18px or 14px bold): contrast ratio ≥ 4.5:1
- Large text (18px+ or 14px+ bold): contrast ratio ≥ 3:1
- UI components and graphical objects: contrast ratio ≥ 3:1

Verify these token combinations specifically:
- `--color-text-secondary` (#8A8A8E) on `--color-surface` (#FFFFFF) — marginal, verify
- `--color-text-muted` (#AEAEB2) — acceptable only for decorative or non-essential text, never for actionable labels
- `--color-accent` (#E8824A) — use for icons and fills only, not body text; white text on accent needs verification
- Status pill text on light backgrounds — verify each combination

Colour must never be the only state indicator. Every status (active, pending, error) has a visible text label alongside the colour.

### Motion

- Does the component respect `prefers-reduced-motion`? The `animations.css` file handles this globally — verify no inline animations bypass it.
- No content flashes more than 3 times per second.

## Tier 2 — Advisory (non-blocking, note for future)

These are logged as findings but do not block completion. They are the foundation for V4/V5 screen reader support. Writing components in a way that makes these easy to add later is the goal.

### ARIA and shadow DOM

Note these without blocking:

- Does the host element have a `role` if it represents a widget? (Will be needed for screen readers — easy to add later if host element is used correctly now)
- Are form inputs using `aria-label` directly rather than relying on external `<label>` associations? (Cross-shadow-boundary `aria-labelledby` does not work — this pattern must be used when ARIA is added)
- Is there a live region (`role="status"` or `role="alert"`) for dynamic content updates? Shadow DOM live regions work in modern browsers (Chrome, Firefox) but may not be announced by all AT tools. Prefer intra-shadow live regions for now; flag for testing with real AT when screen reader support is prioritised in V4.
- Are complex widgets (tab bars, drag buttons, custom selects) structured so ARIA roles can be added to host elements without restructuring shadow DOM?

### Form components (V4/V5 priority)

Form-heavy apps will need full ARIA support. When building form components, note:
- Input/label association pattern to use when ARIA is activated
- Whether the component structure allows `aria-required`, `aria-invalid`, `aria-describedby` to be added to the host element later
- This is the highest-priority ARIA work when screen reader support is requested

## Report format

**Tier 1 findings:**
- ✅ Pass
- ⚠️ Warning — passes but worth monitoring
- ❌ Fail — specific issue, file and line, fix required before completion

**Tier 2 findings:**
- 📋 Note — logged for future ARIA work, not blocking

End with: how many Tier 1 issues (blocking), how many Tier 2 notes (future backlog). Do not fix anything unless explicitly asked.
