# /i18n

Audit a file or module for internationalisation compliance, and scaffold string registration when building new `_lib/` components.

Two tiers of work exist. Only Tier 1 is active until the V4 i18n module is built:

- **Tier 1 — V1 habit (required now).** No hardcoded user-visible strings in `_lib/`. Every string comes from `t()`. English defaults registered via `defineStrings()`. Blocking — a `_lib/` component does not pass `/i18n` without these.
- **Tier 2 — V4 locale system (deferred).** Multi-locale packs, IDB-persisted locale store, fallback chains, `Intl` wrappers, dynamic switching. Non-blocking until the V4 i18n module exists.

---

## Usage

```
/i18n <filepath>           audit a file for hardcoded strings and t() compliance
/i18n locales              audit app/locales/ — check all locale packs have matching keys
```

---

## What counts as a user-visible string

Strings that must go through `t()`:
- Button labels, link text, headings, status messages
- Placeholder text, error messages, confirmation prompts
- Any text a user reads or acts on

Strings that do NOT need `t()`:
- HTML tag names, attribute names, CSS class names
- `customElements.define()` tag names
- IDB object store names, Store action types, event names
- Developer-facing console errors (these are internal, not UI)
- Log entries, test descriptions

---

## Tier 1 audit — `_lib/` files (blocking)

### 1. Scan for hardcoded strings in templates

Read every `template()` method and any string used in a DOM update method. Flag any string literal that is user-visible text — a word or phrase a user reads.

```js
// ❌ Hardcoded — fails
template() {
  return `<button>Reload</button>`;
}

// ✅ Correct — passes
template() {
  return `<button>${t('update.reload')}</button>`;
}
```

### 2. Check imports

Every `_lib/` file that calls `t()` must import from `_lib/core/strings.js`. Flag any file that uses `t()` without this import.

```js
import { t, defineStrings } from '../strings.js'; // ✅ correct relative path
```

### 3. Check default string registration

Every `_lib/` component that uses `t()` must call `defineStrings()` at module scope with English defaults. The registration must happen before any component renders.

```js
// ✅ Correct — registered at module scope, not inside a method
defineStrings('update', {
  available: 'Update available',
  reload:    'Reload',
});
```

Flag if:
- `defineStrings()` is called inside a method or lifecycle hook (too late — may render before registration)
- The namespace is not unique (two components registering the same namespace is a collision risk — flag it)
- Any key registered in `defineStrings()` is not actually used in `t()` calls, or vice versa

### 4. Check `app/` files for `_lib/` string overrides

`app/` files are not required to use `t()`. But if they call `defineStrings()`, check that the namespace and keys match what `_lib/` registered. A mismatched key silently falls through to the key name — flag any mismatch found.

---

## Tier 1 audit — `app/` components

`app/` components do not have the same restriction. They may use hardcoded strings if the app is single-language. However, if `app/locales/` exists, flag any hardcoded string that should be in the locale file.

---

## Tier 2 — V4 locale system audit (`/i18n locales`)

Run this only once `app/locales/` exists with at least two locale files.

### Key parity check

Every key in every locale file must exist in `en.js` (the base locale). Every key in `en.js` must exist in every other locale file. Report:

```
Missing in fr.js: goal.add, goal.complete
Extra in fr.js (not in en.js): goal.archive   ← likely a renamed key
```

### Intl usage check

If the app uses raw `.toLocaleDateString()`, `.toLocaleString()`, `.toLocaleTimeString()`, or `new Intl.*` directly (not via the `_lib/modules/i18n/intl.js` wrappers), flag them. When the V4 i18n module exists, these must go through the wrapper so they respond to locale changes.

## Scaffolding a new `_lib/` component with strings

When building a new `_lib/` component that needs user-visible text:

1. Choose a short, unique namespace (e.g. `'update'`, `'swManager'`, `'toast'`)
2. List every user-visible string the component will show
3. Register them at module scope in English
4. Use `t('namespace.key')` in `template()` and any targeted DOM update

Template:

```js
import { t, defineStrings } from '../../strings.js';

defineStrings('myComponent', {
  title:   'Title text here',
  confirm: 'Confirm',
  cancel:  'Cancel',
});

// In template():
// t('myComponent.title'), t('myComponent.confirm'), etc.
```

---

## Report format

**For file audits:**
- ✅ Pass — no hardcoded strings, `t()` and `defineStrings()` used correctly
- ⚠️ Warning — string is borderline (e.g. a proper noun or brand name) — flag for developer decision
- ❌ Fail — hardcoded user-visible string, missing import, or missing `defineStrings()` — must fix before completion

**For locale audits:**
- List missing keys per locale file
- List extra keys per locale file (may be renamed keys)
- Count total keys in base locale

**Tier 2 notes:**
- 📋 Note — Intl usage outside wrappers, noted for V4

End with: how many Tier 1 failures (blocking), how many warnings, how many Tier 2 notes.
