# Accessibility

Socle takes a tiered approach to accessibility. Tier 1 is required on every component. Tier 2 is deferred until user feedback requests it.

## Tier 1 — required

Tier 1 covers the practices that cost almost nothing and benefit all users:

- **Semantic HTML.** Use `<button>`, `<nav>`, `<main>`, `<article>`, `<h1>`–`<h6>` correctly. Do not use `<div>` where a semantic element exists.
- **Page landmark.** Every page component's `template()` must contain a `<main>` element. Assistive technologies use it to skip to page content.
- **Keyboard navigation.** Every interaction must be reachable and operable via keyboard. Every gesture has a keyboard equivalent (see [Gestures → Keyboard alternatives](gestures.md#keyboard-alternatives)).
- **Touch targets.** All interactive elements have a minimum size of `--touch-target` (40px). Primary actions use `--touch-target-lg` (52px).
- **Colour contrast.** `--color-text-primary` on `--color-bg` passes WCAG AA. `--color-text-secondary` is used for large text only. `--color-text-muted` is decorative only — never use it for actionable labels.
- **Colour not the only indicator.** Every state badge, pill, or status indicator pairs colour with a text label.
- **Motion preferences.** If `core/styles/animations.css` is included, all transitions respect `prefers-reduced-motion` automatically.

The `/a11y` command enforces Tier 1 as blocking — a component is not done until all Tier 1 checks pass.

## Tier 2 — advisory

Tier 2 covers ARIA and screen reader support. Shadow DOM makes ARIA non-trivial — `aria-labelledby` does not cross shadow boundaries, and `role` semantics require careful testing with real assistive technology.

Tier 2 items are logged as advisory notes by `/a11y` but do not block completion. They are prioritised when user feedback requests them.

Current Tier 2 deferrals:

- ARIA roles and live regions for dynamic updates
- `aria-label` and `aria-describedby` on custom controls
- Screen reader announcements for gesture outcomes
- Form label association across shadow boundaries

## The `/a11y` command

Run after building a component, before `/review`:

```
/a11y app/components/my-card/my-card.js
```

The command reads the component's `template()` and checks it against the Tier 1 list. Tier 1 failures block. Tier 2 items are listed as advisory notes.
